I actually think we can do this with very low risk if we treat it as a series of UI migrations instead of a rewrite.

The biggest rule I'd write at the top of the document is:

Battle math, TODOs, EventManager, BattleManager, ActionManager, EnemyBattleController, and background rendering remain untouched until the new UI is fully operational.

That keeps almost all of the risk out of the overhaul.

Overall Risk Assessment

I actually think this is a low-risk overhaul, despite how different the finished screen will look. The reason is that we're not replacing the battle system—we're replacing the presentation layer around it.

Every phase leaves the game in a runnable state. If something goes wrong, we can stop after any phase, fix it, and continue. There is never a point where we need to "rewrite Battle V2."

I would add one more rule to the project docs because I think it's worth treating as a design principle:

Never remove a legacy UI element until its replacement has been tested through an entire battle (start → combat → victory/defeat → return to world).

That one rule alone will eliminate most of the risk during this overhaul.
