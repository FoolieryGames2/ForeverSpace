# Forever Space — Battle Loadout Upgrade / Augment Item Plan

## Goal

Add a new battle loadout item type called `upgrade`.

These are not normal weapons, shields, consumables, or blueprints.  
They are equipped passive battle modifiers.

The player can equip up to **3 upgrade items total** in battle loadout.

Upgrade items affect specific battle stats through metadata.  
They may affect stats positively or negatively in the future, but first pass should use simple positive effects only.

---

2. Add 3 Upgrade Slots To Battle Loadout

The player battle loadout should support:

"equipped_upgrades": []

Rules:

- Max equipped upgrades: 3
- Upgrade items use the new item type: upgrade
- Any mix of subtypes is allowed for now
- Do not allow more than 3 equipped upgrades
- First pass should probably block duplicate upgrade IDs unless later design allows stacking

Example loadout shape:

"equipped_upgrades": [
	"hull_polarizer",
	"generator_heat_sinks",
	"primary_capacitor"
]
3. Upgrade Items Are Reward-Only

Upgrade items should not be craftable.

Upgrade items should not have blueprints.

Upgrade items should not be produced by blueprint systems.

Rules:

"reward_only": true,
"craftable": false,
"blueprint_allowed": false,
"blueprint_id": "",

Design rule:

Blueprints cannot be made for this item type.
Upgrade items are reward items only.
4. Add Generic Battle Upgrade Metadata

Each upgrade item should carry a simple metadata block for battle stat changes.

Recommended first-pass shape:

"battle_upgrade_meta": {
	"max_hull_bonus": 0,
	"max_energy_bonus": 0,
	"primary_damage_bonus": 0,
	"secondary_damage_bonus": 0,
	"secondary_burst_bonus": 0
}

Important behavior rule:

Upgrades should modify derived battle stats.
They should not permanently overwrite base player stats.

Best safe approach:

On loadout refresh or battle start:
1. Start from base battle stats.
2. Read equipped upgrade items.
3. Sum their battle_upgrade_meta values.
4. Build final derived battle stats.

This prevents repeated stat stacking bugs.

5. First Four Upgrade Items To Add
5.1 Hull Polarizer

Purpose:

Armor upgrade.
Increases max hull.
Does not heal the player.
Does not create hull points.

Suggested item ID:

"hull_polarizer"

Effect:

"battle_upgrade_meta": {
	"max_hull_bonus": 25,
	"max_energy_bonus": 0,
	"primary_damage_bonus": 0,
	"secondary_damage_bonus": 0,
	"secondary_burst_bonus": 0
}

Behavior rule:

When equipped:
- Max hull increases by 25.
- Current hull does not increase.
- This is not a repair.
- This is not healing.

Example:

Before equip:
Current hull: 50
Max hull: 100

After equip:
Current hull: 50
Max hull: 125

If removed:

If current hull is above the new max hull, clamp it down to the new max.
Otherwise leave current hull alone.

5.4 Secondary Ammo Extender

Purpose:

Secondary weapon augment.
Improves secondary burst behavior and secondary damage.

Suggested item ID:

"secondary_ammo_extender"

Effect:

"battle_upgrade_meta": {
	"max_hull_bonus": 0,
	"max_energy_bonus": 0,
	"primary_damage_bonus": 0,
	"secondary_damage_bonus": 5,
	"secondary_burst_bonus": 1
}

Behavior rule:

When equipped:
- Add +1 to secondary burst count.
- Add +5 damage to secondary weapon attacks.
- Do not change the base secondary weapon item permanently.

Important ammo rule:

If secondary burst count increases, ammo cost should naturally increase because each burst TODO spends ammo.
Example:
Base burst_count = 2
Upgrade adds +1
Final burst_count = 3
Total ammo cost becomes 3 if ammo_per_burst is 1.
6. Battle Stat Application Rules

Upgrade effects should be applied carefully.

Required safety rule
Never apply upgrade bonuses by repeatedly adding onto already-modified stats.
Always rebuild derived stats from base values.

Correct pattern:

var final_stats = base_stats.duplicate(true)

for upgrade_id in equipped_upgrades:
	var item = item_db.get(upgrade_id, {})
	var meta = item.get("battle_upgrade_meta", {})

	final_stats.max_hull += int(meta.get("max_hull_bonus", 0))
	final_stats.max_energy += int(meta.get("max_energy_bonus", 0))
	final_stats.primary_damage_bonus += int(meta.get("primary_damage_bonus", 0))
	final_stats.secondary_damage_bonus += int(meta.get("secondary_damage_bonus", 0))
	final_stats.secondary_burst_bonus += int(meta.get("secondary_burst_bonus", 0))

8. Save / Load Requirements

Player save data should preserve equipped upgrades.

Add to player loadout save state:

"equipped_upgrades": []

Rules:

- Save equipped upgrade item IDs.
- On load, validate that each item still exists in item DB.
- Remove invalid/missing upgrade IDs safely.
- Never allow loaded upgrade list to exceed 3.

Validation example:

func sanitize_equipped_upgrades(upgrades: Array) -> Array:
	var clean := []

	for upgrade_id in upgrades:
		var id := str(upgrade_id).strip_edges()
		if id == "":
			continue
		if clean.size() >= 3:
			break
		if not item_db.has(id):
			continue

		var item = item_db.get(id, {})
		if str(item.get("item_type", item.get("type", ""))) != "upgrade":
			continue

		if clean.has(id):
			continue

		clean.append(id)

	return clean

Final Design Lock

Battle loadout upgrades are passive reward-only augment items.

They are equipped in one of three upgrade slots.

They modify battle behavior through metadata.

They do not permanently mutate base weapons or base player stats.

Hull and energy max upgrades do not create or restore hull/energy.

Blueprints cannot be made for upgrade items.

This system should be safe, simple, and expandable.
